## ADR-001: Отдельное приложение для биллинга (не часть web-frontend)

**Статус:** Принято  
**Дата:** 2026-03

### Контекст
Биллинг имеет другую целевую аудиторию (бухгалтеры, администраторы), другие требования
к безопасности (финансовые данные), и может развиваться независимо от мониторинга.

### Решение
`web-billing` — отдельное SPA-приложение, отдельный домен/поддомен
(billing.wayrecall.com), отдельный Vite-проект.

### Альтернативы
1. **Раздел внутри web-frontend** — смешивает аудитории, усложняет навигацию
2. **Монолитный frontend (micro-frontends позже)** — overengineering для MVP

### Последствия
- Независимые релизы billing и monitoring фронтов
- Shared UI-компоненты через npm-пакет при необходимости
- Отдельный RBAC: доступ к биллингу ≠ доступ к мониторингу

## ADR-002: Тот же стек что и web-frontend

**Статус:** Принято  
**Дата:** 2026-03

### Контекст
Нужно минимизировать когнитивную нагрузку при разработке двух фронтендов.

### Решение
React 18 + TypeScript 5 + Vite 5 + Zustand + TanStack Query + Tailwind + Shadcn/ui.
Тот же стек, те же паттерны, та же feature-based структура.

### Последствия
- Один разработчик может работать с обоими приложениями
- Общие утилиты (apiClient, auth) можно вынести в shared package
- Одинаковые build/deploy пайплайны

## ADR-003: Billing Service — отдельный Scala-сервис (PostMVP)

**Статус:** Предварительно  
**Дата:** 2026-05

### Контекст
Биллинг-логика (тарификация, расчёт стоимости, выставление счетов) не должна
быть частью User Service или Admin Service — это отдельный bounded context.

### Решение
При реализации создать `services/billing-service/` на Scala 3 + ZIO 2
с собственной БД-схемой `billing.*` в PostgreSQL.

### Альтернативы
1. **Часть User Service** — нарушает SRP, смешивает пользователей и платежи
2. **Часть Admin Service** — админ = мониторинг системы, не финансы
3. **Внешний SaaS (Stripe Billing)** — vendor lock-in, не подходит для RUB

### Последствия
- Отдельный сервис, отдельная БД-схема, отдельный порт (8099)
- Kafka: потребляет `device-events` для тарификации пробега
- REST API через API Gateway
